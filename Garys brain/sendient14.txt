Post-beta, if you’re building your own working lockbox hardware, then yes-ish: you’ll almost certainly need your own backend + device protocol (an “API,” even if you don’t call it that) to manage OTPs at scale.
But you do not necessarily need the lockbox to “call your API to generate OTPs” the way people instinctively assume. You’ve got a few sane architectures:

1) Cloud-validated codes (simplest to reason about) ☁️🔒
Flow


Your backend generates a one-time code (or token)


Driver enters it on the box (keypad/app)


Box has LTE/Wi-Fi and calls your backend: “Is this code valid for Box123 right now?”


Backend returns yes/no + marks it used


Pros


Easy revocation, logging, analytics


Strong control: “used once” is truly enforced


No time-sync headaches


Cons


Requires connectivity on the box


More infrastructure and ops (congrats, you own a mini bank now)


When to use: if reliability + audit trail matter and you can afford connectivity.

2) Offline OTP (TOTP/HOTP style) (box verifies locally) 📵✅
This is the “Google Authenticator” idea applied to a lockbox.
Option A: TOTP (time-based)


Backend and box share a secret key


Both compute the same rolling code based on time


Box checks locally without internet


Pros


Works without connectivity


Fast, cheap to run


Cons


Time sync matters (drift becomes your enemy)


Harder to do true “one-time” without a server (it’s “valid for 30 seconds,” not “valid once”)


Revocation is trickier (you can rotate secrets, but it’s not instant unless connected)


Option B: HOTP (counter-based)


Shared secret + a counter


Each successful unlock increments counter on the box


Pros


No time sync


Can behave more “one-time”


Cons


Counter desync is a whole genre of pain


Needs careful design to resync safely


When to use: when the box must open even in dead zones, and you can tolerate some complexity.

3) Signed unlock tokens (my favorite for “real product”) ✍️🎟️
Instead of “OTP codes,” use cryptographically signed tokens.
Flow


Backend issues an “unlock ticket” with:


box_id


user/driver ID


start_time / end_time


nonce (unique random)




Backend signs it (Ed25519/ECDSA)


Driver presents it to the box (QR, NFC, Bluetooth payload, etc.)


Box verifies the signature using a public key stored on-device


Pros


Box can verify offline


No shared secret that, if leaked, lets attackers mint codes


Clean permission model: “This ticket is valid from 2:10–2:25 for Box123”


Scales nicely and feels “enterprise”


Cons


You still need a backend to issue tickets


If you want true “one-time” enforcement, you need:


connectivity (to record “used” centrally), or


secure local storage of “used nonces” (limited but doable)




When to use: when you want modern security without constant connectivity.

So do you “need your own API”?
You need your own backend service for:


issuing access (codes/tickets)


linking users to boxes


defining time windows


logging/audit trail


revocations/blacklists


device onboarding and key provisioning


That backend will expose endpoints to:


your app


and sometimes your devices


That’s effectively your API, yes.
But you do not need:


a “generate OTP” endpoint that the box must call,
unless you choose the always-online validation model.



The boring but important security reality checklist
If you want this to not be a clown-car security product later:


Device identity: each box needs a unique ID + key material


Secure key storage: at minimum ESP32 flash encryption + secure boot; ideally a secure element (ATECC608A-type)


Replay protection: don’t allow someone to reuse an old code/ticket


Rate limiting: boxes should lock out after X bad attempts


Time windows: make codes valid only briefly


Audit logs: investors and enterprise clients love logs; criminals hate them



My practical recommendation for Sendient post-beta
If your box may be offline sometimes (cars, garages, apartments):
✅ Signed unlock tickets + optional online “mark used” when available.
If you expect reliable connectivity and want maximum control:
✅ Cloud-validated codes.
If you want cheapest offline MVP and accept tradeoffs:
✅ TOTP (but only if you can keep time sync decent).

Bottom line: post-beta, you’ll need your own backend + device protocol. Not because OTPs are mystical, but because access control is the product, and you can’t outsource your core control layer forever without getting held hostage by someone else’s platform.
